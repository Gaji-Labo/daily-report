# 日報フォーマット定義

日報は以下の 4 セクション構成で、Markdown として生成する。

---

## タイトル形式

```
Daily 振り返り YYYY.MM.DD：ユーザー名 #reflection
```

- 日付は `YYYY.MM.DD` 形式（ドット区切り）
- ユーザー名は esa のスクリーンネームまたは `$USER`
- 末尾に `#reflection` タグを付与する

## フォーマット

```markdown
## 今日やったこと

- 作業項目の簡潔な説明
    - 必要に応じてサブ項目で補足

## 今日学んだこと

- 気づきや学びを箇条書きで記載
    - 具体的な内容をサブ項目で補足

## 次にするべきこと

- 翌営業日以降にやるべきタスク
    - 必要に応じてサブ項目で補足

## 今日の Gaji-Labo Style

- Style #N「スタイル名」
    - 該当する行動と、なぜそのスタイルに当てはまるかの説明

## 案件 KPT

- **Keep**
    - 続けたい良い取り組みの説明
- **Problem**
    - 改善すべき課題の説明

## 個人振り返り

- 他者との関係性やタッグフォースについての振り返り
    - 具体的な連携内容の補足

## 案件振り返り

- 案件で感じた課題
    - 具体的な内容の補足
- 品質向上に貢献できたこと
    - 具体的な行動の補足
```

---

## 生成ルール

### 今日やったこと
- アクティビティを **意味のある単位でグルーピング** する。PR 単位ではなく「機能」や「目的」単位。
- リンクは付けず、簡潔な箇条書きで書く。
- サブ項目で作業の詳細を補足する。

### 今日学んだこと
- 対話 (report-interviewer) で得られたユーザーの言葉を **活かしつつ読みやすく整える**。
- ユーザーが語った内容をそのまま貼るのではなく、第三者が読んで理解できる文にする。
- 技術的な具体性を保つ (「勉強になった」ではなく具体的に何を学んだか)。
- サブ項目で詳細や背景を補足する。

### 次にするべきこと
- 対話で出た予定を最優先で記載する。
- それに加えて、未完了のタスク (Open の PR、In Progress の Jira チケット) から推測できるものを追加する。
- チェックボックスは使わず、通常の箇条書きで書く。
- サブ項目で具体的なアクションを補足する。

### 今日の Gaji-Labo Style
- 対話 (report-interviewer) で提案・合意された Style をそのまま記載する。
- Style の番号とタイトルを見出しに、該当する行動とその理由をサブ項目で説明する。
- 対話で合意された内容を忠実に反映し、勝手に追加・変更しない。

### 案件 KPT
- 対話で提案・合意された Keep と Problem をそのまま記載する。
- Keep と Problem はそれぞれ太字見出しで区分する。
- 合意された内容が無いカテゴリはセクションごと省略する。

### 個人振り返り
- 対話で提案・合意された他者との関係性・タッグフォースに関する振り返りを記載する。
- 具体的に誰とどんな連携ができたかをサブ項目で補足する。
- 合意された内容が無い場合はセクションごと省略する。

### 案件振り返り
- 対話で提案・合意された「案件で感じた課題」と「品質向上への貢献」を記載する。
- 合意された内容が無い場合はセクションごと省略する。

### 全体
- `## 今日やったこと` からスタートする (先頭に `# タイトル` は付けない。タイトルは投稿スクリプトが付与する)。
- 簡潔で読みやすい日本語。
- アクティビティログに無い情報を **捏造しない**。
- 対話で語られなかった情報を推測で書かない。
