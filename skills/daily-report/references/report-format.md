# 日報フォーマット定義

日報は以下の 3 セクション構成で、Markdown として生成する。

---

## フォーマット

```markdown
## やったこと

- **作業グループ名**: 何のために何をしたかが伝わる 1〜2 行の説明
  - [リポジトリ名#番号 タイトル](URL)
  - Jira: [チケットキー] タイトル → ステータス
  - 関連する Slack の議論があればその要約

- **次の作業グループ**: ...

## 気づき・学び

- 対話で得られた技術的な気づきや学び
- レビューや議論から得た知見
- 情報が不足する場合: <!-- ここに気づき・学びを追記 -->

## 次やること

- [ ] 翌営業日にやるべきタスク
- [ ] ブロッカーや相談事項
- 情報が不足する場合: - [ ] (追記)
```

---

## 生成ルール

### やったこと
- アクティビティを **意味のある単位でグルーピング** する。PR 単位ではなく「機能」や「目的」単位。
  - 良い例: 「認証基盤の OAuth2 対応」の下に PR 2 本 + Jira チケット
  - 悪い例: PR を 1 件ずつ並べるだけ
- リンクがあるものは必ず Markdown リンクとして付与する。
- Jira チケットはステータスの変化がわかるように書く (例: `→ Done`)。
- Slack の発言はそのまま引用せず、何について議論したかを要約する。

### 気づき・学び
- 対話 (report-interviewer) で得られたユーザーの言葉を **活かしつつ読みやすく整える**。
- ユーザーが語った内容をそのまま貼るのではなく、第三者が読んで理解できる文にする。
- 技術的な具体性を保つ (「勉強になった」ではなく「state と code_verifier の役割の違いを整理できた」)。
- 情報が不足する場合はプレースホルダーを残す。

### 次やること
- 対話で出た予定を最優先で記載する。
- それに加えて、未完了のタスク (Open の PR、In Progress の Jira チケット) から推測できるものを追加する。
- チェックボックス形式 (`- [ ]`) で書く。

### 全体
- `## やったこと` からスタートする (先頭に `# タイトル` は付けない。タイトルは投稿スクリプトが付与する)。
- 簡潔で読みやすい日本語。
- アクティビティログに無い情報を **捏造しない**。
- 対話で語られなかった情報を推測で書かない (プレースホルダーを残す)。
