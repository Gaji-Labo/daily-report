# 日報作成スキル

GitHub、Jira、Slack、Google Calendar からその日のアクティビティを収集し、対話で所感を引き出してから、esa に日報を投稿する。

## ワークフロー

以下の 6 ステップを順に実行する。途中でユーザーが中止を希望した場合はいつでも止める。

### Step 1: 日付の確認

ユーザーに対象日を確認する。指定がなければ今日の日付を使う。

### Step 2: 調査範囲の確認

アクティビティ収集の前に、各データソースの調査範囲を確認する。

**Cowork 環境（メモリー利用可）**:
1. **メモリーを確認**: 過去の調査範囲設定（対象リポジトリ、Jira プロジェクト、Slack チャンネル等）があれば読み込む。
2. **メモリーに前回の設定がある場合**: ステップを止めずに「〇〇の範囲で調査します。変更があればお知らせください」と宣言し、そのまま Step 3 に進む。ユーザーが変更を申し出た場合のみ反映する。
3. **メモリーに設定がない場合**: 各データソースについて調査対象を尋ねる。回答はメモリーに保存する。

**Claude Code 環境（メモリー不使用）**:
- 環境変数（`TARGET_REPOS`、`SLACK_CHANNELS` 等）が設定されていればそれを使う。
- 環境変数が未設定の場合は、各データソースについて調査対象を尋ねる（例: GitHub の対象リポジトリ、Jira の対象プロジェクト、Slack の対象チャンネルなど）。特に指定がなければデフォルト動作（全体検索またはスキップ）で進める。

### Step 3: アクティビティ収集

以下を **並列** で実行し、結果を集約する。各ソースは利用可能な手段（`gh` CLI、MCP サーバー、コネクタ）のうち使えるものを自動判定して取得する。利用できないソースはスキップし、取得できたものだけで進める。

**GitHub** — 対象日の以下を取得:
- 自分が作成した PR
- 自分がマージした PR
- 自分がレビューした PR
- 自分が作成した Issue
- 自分のコミット
- 自分が書いた PR コメント
- 自分の PR に対する他者からのレビュー（承認・変更要求・行コメント）

取得手段: `gh` CLI（`gh search` + `gh api`）または GitHub MCP サーバーを使い、PR/Issue/commit に加えて PR コメント・レビュー・レビューコメントも取得する。具体的な連携手段は CLAUDE.md の実行環境表に従う。

**Jira** — 対象日の以下を取得:
- 当日アサインされて更新された Issue
- 各 Issue のステータスとコメント

取得手段: Atlassian MCP サーバー (`mcp__atlassian`)、Jira コネクタ、いずれか利用可能なものを使用。

**Slack** — 対象日の以下を取得:
- 自分の発言（チャンネル投稿・スレッド参加）

取得手段: Slack MCP サーバー (`mcp__slack`)、Slack コネクタ、いずれか利用可能なものを使用。

**Google Calendar** — 対象日の以下を取得:
- ミーティング・予定の一覧（タイトル、時間、参加者）

取得手段: Google Calendar コネクタ（Cowork 環境）を使用。利用できない場合はスキップする。

**収集結果の表示**: 何が取れたかをユーザーにフレンドリーに一覧表示する。受け取ったレビューがある場合は、誰からどのような指摘・承認があったかも含める。ミーティングがあった場合はその一覧も表示する。

### Step 4: 対話で所感を引き出す

`daily-report:report-interviewer` エージェントに委譲する。

エージェントに渡すデータ:
- 収集したアクティビティログ（Google Calendar のミーティング情報を含む）
- `references/gaji-labo-style.md` の Gaji-Labo Style 一覧

このエージェントは収集したアクティビティを踏まえて、ユーザーに 1〜2 個ずつ具体的な質問をし、
日報の「気づき・学び」「次やること」セクションに書くべき情報を引き出す。
ミーティングがあった場合は、各ミーティングについて「どんなことが決まったか」「自分はどんな発言・提案をしたか」「次のアクションは何か」を具体的に尋ねる。
対話の中でミーティングでの発言や Slack での気遣いなどコミュニケーション面、他者との協力関係、案件の課題や品質への貢献も引き出し、
最後にユーザーの行動をもとに「Gaji-Labo Style」「案件 KPT（K・P）」「個人振り返り」「案件振り返り」をまとめて提案する。

ユーザーが「done」や空入力で終了できる。

### Step 5: 日報 Markdown の生成

収集データと対話内容をもとに、`references/report-format.md` のフォーマットに従って日報を生成する。

生成後、ユーザーに Markdown 全文を表示し、以下の選択肢を提示:
- **そのまま投稿** する
- **修正点を指示** する (指示に従って再生成)
- **中止** する

### Step 6: esa に投稿

ユーザーが投稿を承認したら、利用可能な手段で esa に投稿する。

- タイトル形式: `Daily 振り返り YYYY.MM.DD：ユーザー名 #reflection`
- 同日の記事が既に存在する場合は上書き更新する
- 投稿後、esa の URL をユーザーに表示して完了

投稿手段: esa MCP サーバーが使える場合はそれを優先し、そうでなければ `scripts/post-to-esa.sh` を使う。
