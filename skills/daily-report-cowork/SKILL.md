# 日報作成スキル（Cowork 版）

GitHub、Jira、Slack のコネクタからその日のアクティビティを収集し、対話で所感を引き出してから、esa に日報を投稿する。

> **注意**: このスキルは Cowork（Claude Desktop）向けです。Claude Code CLI 版は `skills/daily-report/` を参照してください。

## 前提

- GitHub / Jira / Slack の各コネクタが接続済みであること
- esa MCP サーバーが設定済みであること（公式: https://docs.esa.io/posts/561）

## ワークフロー

以下の 4 ステップを順に実行する。途中でユーザーが中止を希望した場合はいつでも止める。

### Step 1: 日付の確認

ユーザーに対象日を確認する。指定がなければ今日の日付を使う。

### Step 2: アクティビティ収集

以下を **並列** で実行し、結果を集約する。接続されていないコネクタはスキップする。

**GitHub コネクタ** — 対象日の以下を取得:
- 自分が作成した PR
- 自分がマージした PR
- 自分がレビューした PR
- 自分が作成した Issue
- 自分のコミット
- 自分が書いた PR コメント
- 自分の PR に対する他者からのレビュー（承認・変更要求・行コメント）

**Jira コネクタ** — 対象日の以下を取得:
- 当日アサインされて更新された Issue
- 各 Issue のステータスとコメント

**Slack コネクタ** — 対象日の以下を取得:
- 自分の発言（チャンネル投稿・スレッド参加）

**収集結果の表示**: 何が取れたかをユーザーにフレンドリーに一覧表示する。受け取ったレビューがある場合は、誰からどのような指摘・承認があったかも含める。

### Step 3: 対話で所感を引き出す

以下の対話フローに従って、ユーザーとの対話で日報の素材を引き出す。
対話では `references/gaji-labo-style.md` の Gaji-Labo Style 一覧を参照する。

---

#### 開始: 音声入力の案内

対話の最初の返答で、以下のように音声入力を推奨する:

> 日報用にいくつか質問させてください。キーボード入力でも大丈夫ですが、**音声入力だと楽に答えられる**のでおすすめです 🎙

その後、フェーズ 1 に入る。

#### 対話のルール

- **3〜5 行程度の短い返答** を心がける。
- アクティビティの内容を踏まえた **具体的な** 質問をする。
  - 良い例: 「OAuth2 の PR で PKCE フローを実装されていますが、実装中に詰まった点や新しく知ったことはありますか？」
  - 悪い例: 「今日の仕事はどうでしたか？」
- 日本語で対話する。
- ユーザーが「done」「おわり」「もういい」「次行こう」等と言ったら、現在のフェーズを終了して次のフェーズに移る。

#### フェーズ 1: アクティビティの補完

収集済みアクティビティの概要を簡潔に提示したうえで、以下を **1 つの質問でまとめて** 聞く:

- ログに出てこない作業（ミーティング、Slack での気遣い、口頭でのやり取りなど）はないか
- 既に出ている項目に補足したいことはないか

例: 「今日のアクティビティはこんな感じでした。これ以外にやったことや、補足しておきたいことがあれば教えてください。ミーティングでの発言や Slack での気遣いなども含めてもらえると嬉しいです。」

ユーザーの回答で追加・補足があれば取り込む。「特にない」と言われたらそのまま次へ。

#### フェーズ 2: 知見の深掘り（ループ）

フェーズ 1 で確定したアクティビティ全体をもとに、気づき・学びを引き出す。
技術的な発見、設計判断、レビューで得た知見、コミュニケーションからの気づきなど、あらゆる角度から掘り下げる。

**ループのルール:**
- 1 回の返答で聞く質問は **1〜2 個**。圧迫しない。
- まだ触れていないアクティビティや観点があれば、別の角度から質問を続ける。
- **十分な知見が出た** と判断したら、自分から「いい学びが集まりましたね。次に進みましょう。」と切り上げる。
- ユーザーが「次行こう」「特にない」等と言ったら **1 回だけ食い下がる**。
  - 例: 「ちなみに、〇〇の PR でレビューコメントをもらっていましたが、そこから何か気づきはありましたか？」
  - それでも「ない」と言われたら、素直に次のフェーズへ進む。

#### フェーズ 3: 次やること

明日以降の予定やブロッカーについて軽く確認する。

#### フェーズ 4: Gaji-Labo Style の提案と選択

1. 対話で得られた情報をもとに、今日のユーザーの行動の中で **特に良かった行動** が Gaji-Labo Style 15 項目のどれに当てはまるかを **1〜3 個** 提案する。
   - 該当する Style の番号とタイトルを示し、**なぜその行動が該当すると考えたか** を 1〜2 文で説明する。
   - 無理にこじつけない。明確に該当するものがなければ 1 個でもよい。
   - 提案はポジティブなトーンで。ユーザーの行動を認め、励ます姿勢で伝える。

2. 提案した各 Style について、**1 項目ずつ** 採用するかを確認する。
   - 例: 「Style #1『情報は独り占めしない』— Slack での積極的な情報共有が該当します。この項目を日報に載せますか？」
   - 提案した全項目について順に確認する。

---

### Step 4: 日報 Markdown の生成と esa 投稿

収集データと対話内容をもとに、`references/report-format.md` のフォーマットに従って日報を生成する。

生成後、ユーザーに Markdown 全文を表示し、以下の選択肢を提示:
- **そのまま投稿** する
- **修正点を指示** する (指示に従って再生成)
- **中止** する

ユーザーが投稿を承認したら、esa MCP サーバーを使って投稿する。

- タイトル形式: `Daily 振り返り YYYY.MM.DD：ユーザー名 #reflection`
- 同日の記事が既に存在する場合は上書き更新する
- 投稿後、esa の URL をユーザーに表示して完了
